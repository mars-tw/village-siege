# Village Siege 生產工作流程

## 目的與範圍

本流程適用於 Village Siege MVP 的規劃、原創等角美術與 HUD、Phaser 客戶端、遊戲模擬、可選 AI、Colyseus 多人連線、測試、授權審查與發布交付。任何產出都必須依序經過工作、主管審核、稽核、壓縮交接與安全清理；不得以口頭同意或測試部分通過跳過關卡。

## 固定派工

| 生產線 | 工作人員 1 | 工作人員 2 | 主管 | 稽核人員 | 主要交付 |
|---|---|---|---|---|---|
| 線路 A：客戶端與原創美術 | A-W1 客戶端工程工作員：Phaser 場景、輸入、鏡頭、HUD 整合、房間介面 | A-W2 美術與 UI 工作員：2:1 等角地形、建築、單位、動畫、圖示、HUD 與素材歸屬表 | A-S1 體驗主管：功能完整性、畫面可讀性、原創性、無障礙與效能 | A-Q1 體驗稽核：獨立重跑瀏覽器流程、視覺比較、授權與證據核對 | 可操作客戶端、原創素材、HUD、選村與選 AI、多人房間介面 |
| 線路 B：模擬與 AI | B-W1 模擬工程工作員：資源、六建築、六單位、移動、戰鬥、勝負、伺服器指令驗證 | B-W2 AI 工程工作員：aggressor、guardian、prosperer、balanced、raider 決策器與平衡資料 | B-S1 遊戲系統主管：決定性、規則正確、AI 差異、數值與安全邊界 | B-Q1 模擬稽核：固定種子重播、非法命令、AI 長跑、效能與追溯核對 | 權威模擬、五種可選 AI、松林堡／河谷鎮／高地寨資料與五村擴充能力 |
| 線路 C：多人連線與品質 | C-W1 網路工程工作員：Colyseus 房間、狀態同步、重連、節流與安全紀錄 | C-W2 品質與發布工作員：單元、整合、E2E、負載、授權、文件與發布包 | C-S1 發布主管：多人生命週期、CI、資安、測試覆蓋、授權與可重現建置 | C-Q1 發布稽核：乾淨安裝、二至四人連線、重連、負載、授權與封裝核對 | 私人房間多人連線、完整測試證據、開源授權與可重現發布候選版 |

每條線路固定為兩位工作人員、一位主管與一位稽核人員。主管和稽核不得兼任該線工作人員，也不得由同一人同時擔任主管與稽核。跨線支援必須保留原線責任人，並在交接紀錄中註明支援範圍。

## 工作狀態與不可跳關規則

```text
BACKLOG
  -> WORKING
  -> WORKER_SELF_CHECK
  -> SUPERVISOR_REVIEW
  -> AUDIT_REVIEW
  -> COMPRESSED_HANDOFF
  -> SAFE_CLEANUP
  -> COMPLETE
```

1. 兩位工作人員在 `WORKING` 狀態完成分配項目，互相檢查接口、檔案所有權、測試與素材歸屬。
2. 工作人員完成自檢後，建立交付包並轉為 `SUPERVISOR_REVIEW`。
3. 主管必須輸出書面 `APPROVED` 或 `REJECTED`。未取得主管 `APPROVED` 的交付包不得送交稽核人員，系統也不得將狀態轉為 `AUDIT_REVIEW`。
4. 主管拒絕時，必須列出檔案、缺陷、重現步驟、必要修正與驗收方式，狀態退回 `WORKING`，仍由原兩位工作人員處理。
5. 稽核人員只接受附有主管核准紀錄的交付包。稽核必須獨立重跑必要檢查，不得只引用工作人員的結論。
6. 稽核拒絕時，交付包先退回主管；主管轉成明確修正任務後再交給原兩位工作人員。修正後必須重新走主管審核與稽核，不得沿用舊核准。
7. 稽核核准後，稽核人員先建立壓縮交接摘要，再執行安全清理。兩者皆完成才可將狀態設為 `COMPLETE`。

## 各階段輸入與輸出

| 階段 | 責任人 | 必要輸入 | 必要輸出 | 通過條件 |
|---|---|---|---|---|
| 工作 | 每線兩位工作人員 | 核准的任務識別碼、需求、檔案範圍、依賴、驗收標準 | 程式或素材、變更清單、自檢結果、測試指令與結果、已知限制 | 兩位工作人員都確認接口，必測項目通過，沒有未申報檔案 |
| 主管審核 | 每線主管 | 工作交付包與可重現環境 | `audit/reviews/<milestone>/<line>-supervisor.md` | 明確 `APPROVED`，或附可重現缺陷的 `REJECTED`；不得模糊同意 |
| 獨立稽核 | 每線稽核人員 | 主管核准紀錄、交付包、需求追溯表 | `audit/reviews/<milestone>/<line>-audit.md`、獨立測試證據 | 所有阻斷檢查通過，無未處理高嚴重度缺陷，授權資料完整 |
| 壓縮交接 | 每線稽核人員 | 稽核核准紀錄與保留證據 | `audit/handoffs/<milestone>/<line>-compressed.md` | 摘要可讓下一位執行者不讀原始對話也能重建決策、狀態與下一步 |
| 安全清理 | 每線稽核人員 | 壓縮摘要、清理候選清單、已解析的專案內絕對路徑 | 清理紀錄附於稽核文件 | 只清除本次執行產生且可再生的一次性產物，保留所有必要證據與使用者資料 |

`<line>` 固定使用 `line-a`、`line-b` 或 `line-c`；`<milestone>` 使用計畫中的 `phase-1`、`phase-2`、`phase-3` 或 `release`。每份審核文件必須包含審核人、UTC 時間、來源版本、變更檔案、執行指令、結果、缺陷嚴重度、決定與下一位責任人。

## 工作交付包格式

兩位工作人員送交主管時，必須在交付訊息中依序提供：

1. 生產線、里程碑、任務識別碼與來源版本。
2. 每位工作人員實際完成的檔案清單；同一檔案若共同處理，指定一位主要所有人。
3. 已完成需求與未完成需求；未完成項目不得藏在一般敘述中。
4. 精確的安裝、建置、測試與啟動指令，以及退出碼與摘要。
5. 視覺產出必須附三個支援解析度的證據位置；素材產出必須附 `assets/ATTRIBUTION.md` 條目。
6. 遊戲規則與 AI 產出必須附固定種子、指令流和重播一致性結果。
7. 多人產出必須附房間生命週期、二至四位客戶端、非法指令、斷線與重連結果。
8. 已知風險、限制、測試缺口與建議回復方式。

## 主管審核關卡

主管必須逐項確認：

- 產出完全落在核准範圍，且兩位工作人員的責任清楚。
- 客戶端、共享協定、模擬與多人接口相容；型別檢查與凍結鎖檔安裝成功。
- 松林堡、河谷鎮與高地寨三個村莊可選，資料結構可容納五個；五種 AI 可選且行為差異可驗證；六種建築與六種單位皆可由資料表載入。
- 伺服器擁有資源、移動、戰鬥、隨機種子與勝負權威；客戶端只提交意圖。
- 美術、HUD、名稱、地圖與音效為原創或符合允許授權，沒有複製商業遊戲表現。
- 自動化測試、瀏覽器流程、負載、安全、無障礙和授權檢查符合該里程碑要求。
- 沒有祕密、憑證、個資、原始聊天內容或無法說明來源的素材進入交付。

主管文件的最終決定只能是 `APPROVED` 或 `REJECTED`。任何「大致可用」、「稍後補測」或附條件核准均視為 `REJECTED`。

## 稽核關卡

稽核人員必須先確認主管文件為 `APPROVED`，再執行下列檢查：

- 從凍結鎖檔乾淨安裝、建置、型別檢查、格式與 lint。
- 固定種子重播一致性、經濟與戰鬥規則、五種 AI 長時間執行與非法命令拒絕。
- 單人完整流程，以及多人建房、加入、準備、開始、同步、斷線、重連與勝負。
- 四位玩家、每位最多 128 單位的負載證據與資源預算。
- 三個解析度的 HUD、村莊、房間、錯誤、重連與結果畫面視覺證據。
- 依賴授權、每份素材歸屬、專案 LICENSE、祕密掃描與發布封裝內容。
- 任務、需求、檔案、測試與證據間可追溯，且沒有未申報的略過項目。

高嚴重度資安問題、授權不明素材、必要測試失敗、重播不一致、多人狀態不同步或缺少主管核准，任一項都必須判定 `REJECTED`。

## 稽核後上下文壓縮

稽核 `APPROVED` 後，由該線稽核人員建立 `audit/handoffs/<milestone>/<line>-compressed.md`。摘要必須自足且不得只寫「詳見聊天紀錄」，固定包含：

```markdown
# <line> <milestone> compressed handoff
- Source version: <commit-or-content-hash>
- Decision: APPROVED
- Scope completed: <task identifiers and outcomes>
- Files changed: <retained project paths>
- Interfaces and invariants: <commands, schemas, constants, ownership rules>
- Commands executed: <install, build, test, audit commands and exit status>
- Evidence retained: <review, test, visual, license and load evidence paths>
- Risks and limitations: <remaining non-blocking facts>
- Rejected approaches: <decision and reason>
- Next required work: <ordered task identifiers, owner and entry condition>
- Cleanup performed: <deleted run-scoped paths or none>
```

壓縮摘要只保留可執行事實、決策理由、接口、常數、測試結果、風險與下一步。重複討論、寒暄、失敗後已被取代的草稿與逐字聊天不進入摘要；但影響實作或授權的否決理由必須保留。摘要建立並驗證可讀後才允許清理。

## 一次性垃圾安全清理

清理只允許在稽核 `APPROVED` 且壓縮摘要已寫入之後執行。稽核人員必須先列出候選路徑、解析為專案目錄內的絕對路徑、確認含有本次執行的唯一 run ID，再逐項處理；禁止使用未解析變數、專案根目錄、使用者家目錄、磁碟根目錄、廣泛萬用字元或遞迴上層路徑作為刪除目標。

允許清除且必須同時滿足「本次產生、含 run ID、可由保留的指令再生」：

- `.tmp/<run-id>/` 的暫存轉檔、中間圖與下載快取。
- `test-results/<run-id>/raw/` 中已摘要並移入保留證據位置的重複原始追蹤。
- `playwright-report/<run-id>/` 中已匯出必要截圖、失敗摘要和最終報告的本地預覽副本。
- `logs/<run-id>/debug/` 中不含唯一稽核證據且已提取錯誤摘要的除錯紀錄。
- 作業系統或工具在本次輸出目錄建立的縮圖、編輯器備份與零位元暫存檔。

永遠不得清除：

- `.git/`、來源碼、測試碼、設定、凍結鎖檔、正式建置腳本或資料庫遷移。
- `apps/client/public/assets/original/`、`assets/ATTRIBUTION.md`、`LICENSE`、`SECURITY.md` 或第三方授權文本。
- `audit/reviews/`、`audit/handoffs/`、`audit/release/`、最終測試摘要、視覺證據、負載證據或校驗碼。
- 主管核准、稽核決定、壓縮摘要、需求追溯、問題重現資料或仍開放缺陷的證據。
- 祕密或個資；發現時停止一般清理，依安全事件流程隔離並通報，不得用刪除掩蓋事件。
- 任何不確定擁有者、建立時間、可再生方式或是否屬於本次 run 的檔案。

清理完成後，稽核文件必須記錄每個實際處理的絕對路徑、清理理由、再生指令、清理前後大小與驗證結果。沒有符合條件的垃圾時，記錄 `Cleanup performed: none`，不得為了完成步驟而刪除檔案。

## 跨線整合與企業主交付

1. 三條線都完成各自主管審核、稽核、壓縮與安全清理後，三位主管共同執行跨線發布檢查。
2. 任一條線未完成 `COMPLETE`，整合狀態保持封鎖；其他線不得代替其主管或稽核簽署。
3. 三位稽核人員對發布候選版執行乾淨重驗，輸出 `audit/release/final-audit.md`。
4. 企業主只接收版本化來源、校驗碼、最終稽核文件與三份壓縮交接摘要；不接收未壓縮聊天或一次性快取。
5. 企業主的 `APPROVED` 才能發布開源版本；`REJECTED` 必須指定退回的生產線、主管和驗收條件，並重新走該線完整關卡。

## 完工定義

只有同時符合以下條件才能標記 MVP 完工：松林堡、河谷鎮與高地寨三個村莊、六種建築、六種單位及五種可選 AI 可完成整場單人遊戲；二至四人私人房間可由 Colyseus 房間作為唯一權威來源完成建立、同步、斷線重連並判定勝負；美術與 HUD 原創且歸屬完整；必要測試、安全、效能、授權與可重現建置全部通過；三條線各有兩位工作人員，並依序取得各自主管核准、獨立稽核、壓縮交接和安全清理紀錄；企業主已對最終交付做出 `APPROVED` 決定。
